## Implementation Finding: LayoutCustomizationsProvider Context Crash

**Date**: 2026-04-08
**Status**: Open — fix identified, not yet applied
**Commit**: `6d862e11` (feat: decouple sidebar from task selection)
**Severity**: Critical — app crashes on load

### The Error

```
useLayoutCustomizations must be used within a LayoutCustomizationsProvider
```

App crashes immediately on open.

### Root Cause

`useCardDetailLayout` (in `resize/use-card-detail-layout.ts`) internally calls `useLayoutResetEffect` → `useLayoutCustomizations`, which requires `LayoutCustomizationsContext` from an ancestor `LayoutCustomizationsProvider`.

**Before the refactor**, `useCardDetailLayout` was called inside `CardDetailView`, which was rendered as a child of `LayoutCustomizationsProvider` in App.tsx's JSX:

```
App.tsx renders:
  <LayoutCustomizationsProvider>        ← provider here
    ...
    <CardDetailView>                    ← child component
      useCardDetailLayout()             ← hook called INSIDE provider tree
        useLayoutResetEffect()
          useLayoutCustomizations()     ← context available ✓
```

**After the refactor**, `useCardDetailLayout` was moved to `App.tsx` itself so tab state (activeTab, handleTabChange, etc.) could be shared between the Home side panel and CardDetailView. But `LayoutCustomizationsProvider` is rendered in App.tsx's own JSX return. React hooks run at the component level — they cannot access context from a provider that the same component renders:

```
App.tsx:
  useCardDetailLayout()                ← hook runs at App level, BEFORE render
    useLayoutResetEffect()
      useLayoutCustomizations()        ← NO provider ancestor → CRASH

  return (
    <LayoutCustomizationsProvider>     ← provider is App's CHILD, not ancestor
      ...
    </LayoutCustomizationsProvider>
  )
```

A component can only consume context from providers that are **above** it in the component tree, not from providers it renders in its own JSX.

### Why This Wasn't Caught

**1. TypeScript can't verify context availability.** TypeScript checks that `useLayoutCustomizations()` returns the correct shape. It cannot verify that a React context provider exists at the right level of the runtime component tree. Context availability is a runtime invariant, not a type constraint.

**2. Unit tests mock `useLayoutResetEffect`.** Both `card-detail-view.test.tsx` and the layout test environment use:
```typescript
vi.mock("@/resize/layout-customizations", () => ({
  useLayoutResetEffect: () => {},
}));
```
This replaces the real `useLayoutResetEffect` (which calls `useLayoutCustomizations`) with a no-op. The mock is correct for isolated unit tests, but it means the context requirement is never exercised in tests.

**3. No integration test renders `<App />`.** The test suite has unit tests for individual components and hooks, but no test renders the full App component. An App-level render test would have caught this immediately because the real `useLayoutCustomizations` would throw on first render.

**4. The spec didn't flag the hook→provider ordering constraint.** The spec correctly identified that `useCardDetailLayout` needed to move from `CardDetailView` to `App.tsx`, and that `LayoutCustomizationsProvider` was in `App.tsx`. But it didn't model the constraint that a hook called at component level cannot consume context from a provider that same component renders. The spec's "Dependencies that must move" section listed `useCardDetailLayout` but focused on what data it needs, not on what context providers it requires.

**5. Build succeeds.** `npm run build` passes because the code is syntactically and type-theoretically correct. The error is purely a React runtime invariant violation — no compile-time signal exists.

**6. Pre-commit tests pass.** The pre-commit hook runs `biome check`, `tsc --noEmit`, and `vitest run test/runtime test/utilities` (runtime tests only, not web-ui tests). Even if web-ui tests ran, they mock the context dependency (see point 2).

### Fix Options

**Option A: Split App into outer wrapper + inner component**

Create a thin wrapper that renders `LayoutCustomizationsProvider`, and move App's hook-heavy logic into an inner component that is a child of the provider:

```typescript
export default function App(): ReactElement {
  const { resetBottomTerminalLayoutCustomizations } = useTerminalPanels(...);
  return (
    <LayoutCustomizationsProvider
      onResetBottomTerminalLayoutCustomizations={resetBottomTerminalLayoutCustomizations}
    >
      <AppContent />    {/* useCardDetailLayout called here, inside provider */}
    </LayoutCustomizationsProvider>
  );
}
```

Challenge: `useTerminalPanels` (which provides `resetBottomTerminalLayoutCustomizations` for the provider) depends on `selectedCard`, `workspaceGit`, etc. The split point needs careful routing of which hooks go in outer vs inner component.

**Option B: Remove `useLayoutResetEffect` from `useCardDetailLayout`** (simpler)

Move the layout reset effect out of `useCardDetailLayout` and into a component that is guaranteed to be inside the provider (e.g., `CardDetailView`). `useCardDetailLayout` would export a `resetLayoutRatios` callback, and the consuming component would wire it to `useLayoutResetEffect`:

```typescript
// useCardDetailLayout — no more useLayoutResetEffect call
return { ...existing, resetLayoutRatios };

// CardDetailView (inside the provider tree):
useLayoutResetEffect(resetLayoutRatios);

// App.tsx also needs it for the home side panel ratio —
// could add useLayoutResetEffect in a small wrapper inside the provider.
```

This keeps `useCardDetailLayout` free of context dependencies, making it safe to call from any tree level.

## Implementation Finding: Browser Tab Not Showing Project Name

**Date**: 2026-04-08
**Status**: Open — not yet investigated in detail
**Commit**: `6d862e11` (feat: decouple sidebar from task selection)
**Severity**: Minor — cosmetic, no functional impact

### Symptom

The browser tab no longer displays the project name after the sidebar refactor.

### Likely Cause

Before the refactor, `ProjectNavigationPanel` was rendered as a standalone `<aside>` directly in `App.tsx` whenever no task was selected (`!selectedCard`). It was always mounted in the home view.

After the refactor, `ProjectNavigationPanel` is rendered inside the Home tab's side panel slot, which only mounts when `activeTab === "home"`. If `activeTab` is `null` (side panel collapsed) or a task-tied tab, the `ProjectNavigationPanel` unmounts entirely.

If there is a `document.title` effect or similar hook that depends on `ProjectNavigationPanel` being mounted — or if the project name display is derived from something in that component's render tree — it would stop working when the panel is unmounted.

### Investigation Path

1. Search for `document.title` usage across the codebase — identify what sets the browser tab title and what data it reads.
2. Check if the title is set by a hook inside `ProjectNavigationPanel` or by something at the `App.tsx` level that reads project state.
3. Check if the title depends on `navigationCurrentProjectId` or `currentProjectId` — these are App-level state that should be unaffected by the refactor.
4. If the title is set by App.tsx or a hook called from App.tsx, the issue may not be related to ProjectNavigationPanel at all — could be a side effect of the `LayoutCustomizationsProvider` crash (finding above) preventing the app from rendering far enough to set the title.

### Relationship to Provider Crash

This may not be a separate bug. If the `LayoutCustomizationsProvider` crash (finding above) prevents the app from rendering at all, the browser tab title would never be set. Fix the provider crash first, then re-test whether the tab title issue persists independently.
